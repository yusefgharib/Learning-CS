Description

This is the final stage of the project where we will add some finishing touches to the game. The game is not very challenging at the moment: let's add enemy encounters! How do you know if you're going to encounter an enemy at a given location? Well... you don't. You can buy an upgrade to see the probability of encountering one. Every enemy encounter means one of your robots had to stay back and fight the enemy, so the player loses it, but the remaining robots are still able to get away with the loot. The game ends when you lose all of your robots. You can also buy an upgrade to see how much titanium you will earn from each found location. Of course, you can also buy more robots if you can afford them.

We will also polish our game by adding high scores stored in a file and a help screen with information about the game and the controls.

Optional after project completion

After you complete this project, you are free to keep experimenting with changing/adding different game mechanics to make the game more interesting and enjoyable. You might want to experiment with the encounter rate to make the game more balanced, or you could go even further and implement a battle system, giving your robots and the enemy HP, attack and defense stats, interactive battles, etc. As they say, the sky is the limit!

Objectives
  1. Now each location should also have an encounter rate between 0% and 100%. The higher the probability, the more likely that an enemy encounter is going to happen. After the project, you can experiment with how this value is set to make the game better. For now, we will give you a simple option to implement.
    1. During the searching, right after generating the place name and titanium amount, randomly generate the encounter rate (a number between 0 and 1) using random.random().
    2. If the encounter rate is visible, display that number (encounter rate) as a percentage rounded to the nearest whole number, for example, 0.6843... --> 68%. For the calculations, however, keep the original unrounded number.
    3. If the player chooses to explore a certain location, generate a second random number between 0 and 1 (using random.random()).
    4. If the second number is smaller than the first number, then the Encounter happens; if the second number is larger or equal to the first, then there is no encounter.
  2. After each enemy encounter, one robot is lost. Make sure those changes are reflected in the HUB after each encounter.
  3. The upgrade shop should include the following upgrades:
    1. Titanium Scan for 250 titanium — shows titanium amount the player will obtain upon exploring a location. Displays next to location name for all searches.
    2. Enemy Encounter Scan for 500 titanium — shows enemy encounter rate (rounded to the nearest percent) next to the location name in all searches.
    3. New Robot for 1000 titanium — essentially gives the player an extra life. Adds an extra robot (exactly the same as the previous ones) to the hub.
    The purchase is only possible if a player has enough titanium to buy the upgrades. Make sure these upgrades don't disappear after saving and loading the game. The player selects the upgrade they want to buy by entering the number corresponding to that upgrade, which should be the same as the upgrade's position in the list above.
  4. If all the robots are lost, the game is over, and a "Game Over" message should be displayed on the screen. Remember that when all robots are lost, titanium is not obtained from that final (failed) exploration.
  5. The score (titanium balance) along with the player's name is then saved to a file containing high_scores in its filename if it is among the 10 highest scores so far. How you represent the data in the file is again completely up to you, just make sure the data persists when the game is reloaded.
  6. The top 10 scores and the players' names should be displayed on the high scores screen which can be accessed from the main menu. If there are less than 10 scores, only display the existing ones, e.g. if the game was played (and game over was reached) 6 times only show 6 scores with names. You should structure the high scores screen as follows: 
    
      HIGH SCORES

      (1) name1 8073
      (2) name2 8023
      (3) name1 567
      (4) name3 127
    If two players obtain the same score, the player who obtained that score first is placed higher on the leaderboard. If a player with the same name played the game multiple times, keep record of all of their scores that made it onto the top 10 high scores.
  7. As a finishing touch, add some help text (accessed from the main menu) to tell the player what the game is about and how to control the game.

Note: for this stage, please have animation turned off by default.

The greater-than symbol followed by a space (> ) represents the user input. Note that it's not part of the input.
